Тема: Composite — дерево як єдина колекція

Вступ
Патерн Composite (Компонувальник) — це структурний патерн проєктування, який дозволяє згрупувати об'єкти у деревоподібні структури та працювати з цією структурою так, ніби це один одиничний об'єкт.

1. Як це працює: єдиний інтерфейс для всіх
Суть патерну полягає в тому, що і прості елементи (Leaf), і складні контейнери (Composite), які містять інші елементи, реалізують один і той самий інтерфейс.

Клієнтський код (та частина програми, що викликає методи) не повинен перевіряти, з чим він працює — з одним об'єктом чи з цілою гілкою дерева.

Якщо виклик йде до листа (простого об'єкта), він просто виконує дію.

Якщо виклик йде до контейнера (вузла), він передає команду всім своїм дочірнім елементам, збирає результати та повертає підсумок.

Таким чином, складна ієрархія перетворюється для клієнта на "єдину колекцію", де кожен елемент реагує на однакові команди.

2. Реальні сценарії використання
Сценарій А: Файлова система
Це класичний приклад. У вас є файли (листки) та папки (контейнери).

Папка може містити файли та інші папки.

Якщо ви хочете дізнатися розмір папки, ви викликаєте метод getSize(). Папка пробігається по всіх вкладених файлах, сумує їхній розмір і повертає результат. Для користувача процес підрахунку розміру одного файлу і цілої папки виглядає однаково.

Сценарій Б: Графічний інтерфейс (GUI)
Елементи інтерфейсу часто вкладені один в одного.

Є прості елементи: Кнопка, Текстове поле (Leaf).

Є контейнери: Панель, Вікно (Composite).

Коли потрібно перемалювати вікно, викликається метод draw(). Вікно викликає draw() у Панелі, а Панель — у всіх кнопок всередині неї.

Сценарій В: Меню замовлення / Чеки
Уявіть структуру товару.

Простий товар: "Ноутбук".

Комплексний товар (коробка): "Набір геймера" (містить Ноутбук, Мишку, Навушники).

Метод getPrice() для коробки просто сумує ціни всіх компонентів всередині. Клієнт може купувати як окремі товари, так і набори, не змінюючи код підрахунку ціни.

3. Переваги та недоліки
Переваги:

Спрощення клієнтського коду: Клієнту не треба знати структуру об'єкта (чи це проста сутність, чи складна). Не потрібно писати перевірки if (object is Group).

Легке розширення: Можна легко додавати нові види елементів у дерево, не ламаючи існуючий код.

Універсальність: Дозволяє будувати складні рекурсивні структури.

Недоліки:

Занадто загальний інтерфейс: Доводиться створювати методи, які можуть не мати сенсу для деяких елементів (наприклад, метод addChild для "Листка", який не може мати дітей).

Складність обмежень: Важко обмежити типи об'єктів, які можна додавати в певний контейнер (на рівні типізації), це доводиться робити через перевірки під час виконання.

Висновок
Composite ідеально підходить для ситуацій, де об'єкти повинні формувати ієрархію "частина-ціле", і ви хочете, щоб клієнт взаємодіяв з ними однаково. Це робить систему гнучкою та легкою для масштабування.
